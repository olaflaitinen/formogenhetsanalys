## Maintainership

### Becoming a Maintainer
- Significant contributions to the project
- Understanding of the codebase
- Commitment to long-term maintenance
- Agreement with EUPL-1.2 terms

### Maintainer Responsibilities
- Review and merge pull requests
- Release management
- Issue triage
- Security response
- Roadmap planning

## Decision Making

### Technical Decisions
- Made by maintainer with community input
- Documented in issues and pull requests
- Reversible when possible
- Consensus preferred for major changes

### Governance Decisions
- Changes to license require unanimous maintainer agreement
- Changes to governance require community consultation
- Deprecation requires 6-month notice
- Breaking changes require major version increment

### Security Policy
See SECURITY.md.

- Vulnerability disclosure process

- 90-day disclosure deadline

### Security Response
- Maintainer team responds within 48 hours
- Fixes prioritized over features
- Security releases out-of-cycle if needed

## Release Process

### Versioning
- Semantic versioning (SemVer)
- MAJOR: Breaking changes
- MINOR: New features (backward compatible)
- PATCH: Bug fixes (backward compatible)

### Release Criteria
- All tests passing
- Documentation updated
- CHANGELOG.md updated
- Security scan clean
- REUSE compliance verified

### Release Steps
1. Create release branch
2. Update version in `_version.py`
3. Update CHANGELOG.md
4. Create GitHub release
5. Publish to PyPI
6. Tag in Zenodo
